# Agent Stem

This write-up explains, in my own words, how I approached this task. Broadly, the process had three stages: first the research phase, then the (vibe) coding phase, and finally refinement and testing.

## Research Phase

This task around the stem agent operates at a meta level: it is a system capable of building systems. That makes it fundamentally different from most agent architectures. The typical agent setups today — at least the ones I have worked with — are ReAct (Reason + Act), DAG workflows, and multi-agent systems. However, none of these architectures truly evolve themselves. For example, neither ReAct nor DAG workflows mutate or improve over time. This suggested that the agent needed a completely different approach.

My initial intuition was therefore to build some kind of evolutionary agent. Based on both my academic and professional experience, this problem felt very similar to systems that benefit from iterative optimization and mutation. After going back and forth with several LLMs — mainly Gemini and Claude — I arrived at the final design for the stem agent.

### AgentSpec (the core)

`AgentSpec` is the core object of the architecture. It is loosely inspired by the AgentSpec framework:  
https://oracle.github.io/agent-spec/development/agentspec/index.html

It is a flat, JSON-serializable dataclass that encodes every architectural decision about a candidate agent: its `system_prompt`, `architecture_type` (linear chain, reflection loop, multi-step search, or adversarial QA), `search_strategy`, `synthesis_strategy`, `max_search_rounds`, `self_critique`, `citation_style`, and `tools`.

It also contains a `mutation_log`: an append-only audit trail of every change made by the differentiation loop, including the reasoning behind each mutation. This is the artifact that evolves across phases and is ultimately written to disk at the end of execution.

More details can be found in:

`stem_agent/core/agent_spec.py`

## Design Decisions

The final agent operates in five phases:

### Phase 0: Ground Truth

Before the agent can evaluate itself, it needs a reliable set of questions and reference answers grounded in real evidence — not generated by the same LLM that will later be evaluated. Phase 0 gathers evidence from four sources (DuckDuckGo, arXiv, Semantic Scholar, and Wikipedia), injects it into a prompt, and asks the LLM to generate 15 questions distributed across three difficulty tiers:

- 5 easy
- 6 medium
- 4 hard

Each question includes a `ground_truth` answer and a list of `key_facts` that the generated answer must contain. The result is cached to disk so subsequent runs can reuse it without re-fetching the sources.

### Phase 1: Sense

This stage essentially asks the LLM:

> “How is this class of problems typically solved?”

The result is a structured JSON map of domain knowledge. More details can be found in the code, but conceptually this phase provides the foundational information used to construct the agent.

### Phase 2: Hypothesize

In this stage, the first agent specification (`v0`) is produced. The dataclass behind the agent is instantiated, effectively creating the initial genome of the system. As explained earlier, the `AgentSpec` is the heart of the architecture, and this is where its first version comes into existence.

### Phase 3: Differentiate

The agent is then put to the test and evaluated to determine whether it is underperforming. Evaluation is performed using an LLM-as-judge across four metrics:

| Dimension | What it measures |
|---|---|
| `factual_accuracy` | Are the facts correct? Do the required ground-truth facts appear? |
| `coverage` | What fraction of the required key facts did the answer include? |
| `coherence` | Is the answer well-structured and readable? |
| `source_diversity` | Does it synthesize multiple perspectives? |

Each metric is scored from 0 to 1, and the mean is computed across all questions. The score for an iteration is the composite mean across the 15-question benchmark.

The loop runs for a maximum of 8 iterations. Early stopping is triggered if:

- at least 3 iterations have completed, and
- the improvement delta in the last 2 iterations is below `0.02`.

After each iteration, the best-performing version of the agent is retained. That becomes the final evolved agent.

### Phase 4: Crystallization

No LLM calls occur during this phase. The winning specification from Phase 3 is written to `final_agent.json`.

A before/after comparison is written to `eval_results.json`, containing:

- the aggregate scores for `v0` and the final version,
- the delta between them, and
- the complete `mutation_log`.

A per-tier breakdown is also written to `eval_by_tier.json`, showing whether improvements were concentrated on easy, medium, or hard questions.

Finally, the phase prints a one-screen summary to stdout.

### Experiments

Initially, I ran the system without the Ground Truth phase and with more generic questions. The differentiation loop completed all 8 iterations and appeared to converge, but the final agent was not meaningfully better than `v0`.

Heuristically speaking, the final `system_prompt` became poisoned: the LLM was both generating the evaluation questions and judging its own answers, which meant the evaluation signal carried very little real value.

This made the creation of the Ground Truth phase both necessary and ultimately the most impactful change in the project. Once Phase 0 grounded the questions in external evidence from arXiv and the web, the composite score at `v0` dropped noticeably — the benchmark became harder and the baseline more honest. More importantly, the improvement delta produced by the differentiation loop became meaningful. The before/after gap widened from nearly zero to a consistent improvement across all difficulty tiers.

Finally, I conducted an experiment comparing three models:

- GPT-5.4-mini
- GPT-5.4
- GPT-5.5

GPT-5.5 achieved the best performance on code review — the only domain where all three models were evaluated — across the mean of the four evaluation metrics. GPT-5.4 performed the worst overall. Surprisingly, GPT-5.4-mini landed in the middle on the shared benchmark and delivered by far the best performance-per-euro ratio.

More details can be seen in the [experiments.md](experiments.md) file.

### What Surprised Me

The LLM-as-mutation-proposer worked significantly better than I expected. I initially assumed that the introspect → mutate loop would generate random or contradictory modifications. In practice, however, it consistently identified the correct weaknesses. For example, if the answers lacked depth, it would propose increasing `max_search_rounds` or switching from a `linear_chain` architecture to a `reflection_loop`.

The resulting `mutation_log` reads almost like a human debugging session, with coherent reasoning connecting one version to the next.

The second surprise was how much evaluation design mattered relative to architecture design. I spent most of the early development time refining the differentiation loop, but the biggest quality improvement came from fixing the evaluation setup itself — grounding questions in external evidence and separating question generation from answer generation.

A strong evaluation pipeline is load-bearing; without it, the rest of the system is effectively optimizing against noise.

The third surprise was the performance of GPT-5.4-mini. Although GPT-5.5 achieved the best raw scores, it is also a reasoning-heavy model with significantly higher cost. In a realistic production environment, the mini model may actually be the preferable choice because of its efficiency-to-cost ratio.

---

### What Failed

Token budget failures were the most frequent technical blocker. Values such as `max_tokens=512` may feel generous for normal chat responses, but they are completely inadequate for structured JSON outputs containing multiple nested string fields.

Before increasing the token budgets and introducing LangChain’s `JsonOutputParser`, nearly every phase would eventually fail with errors such as:

`JSONDecodeError: Unterminated string`

The parser helped standardize fence stripping and truncation recovery, while the token limit increase for GPT-5.5 solved several issues caused by the stricter 2048-token cap used with other models.

The deeper failure, however, was the circular evaluation problem described earlier. In the first version of the system, there was no Ground Truth phase, meaning the same LLM both generated and evaluated the questions. The differentiation loop appeared to converge, but in reality it was merely fitting to its own biases.

This was not an immediately obvious bug. It only became clear after inspecting the final `system_prompt` and noticing that the system had effectively optimized itself toward vagueness rather than depth.

### What I Would Do With More Time

The two biggest gaps are:

1. real tool usage
2. population-based search

At the moment, `multi_step_search` performs multiple LLM calls, but it does not actually query the web between rounds. The iterations deepen only on parametric knowledge. Integrating a live search tool between rounds and feeding the retrieved evidence back into the context would make the architecture fundamentally different from a simple `reflection_loop` and likely much stronger on difficult questions.

On the optimization side, the differentiation loop currently behaves as a single-trajectory hill-climbing process: one specification mutates in one direction at a time.

A more sophisticated approach would run 3–5 parallel specifications from Phase 2 onward, evaluate all of them, and apply crossover operations between the best-performing traits. For example:

- taking the `search_strategy` from one agent,
- the `architecture_type` from another,
- and combining them into a new candidate.

This would explore the specification space far more efficiently.

I would also separate the judge model from the candidate model to reduce self-serving bias, and add per-tier convergence criteria so the loop continues running while hard questions are still improving, even if easy and medium questions have already plateaued.
